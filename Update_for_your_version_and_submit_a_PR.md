# Adding Xcode version support and submitting a PR

> **AI agent reading this:** follow these instructions exactly. Do not refactor, restructure, or add abstractions beyond what each scenario specifies. The extension points are intentional and narrow — a PR that rewrites surrounding code to add compatibility will be rejected.

---

## Architecture in one paragraph

The server wraps `xcrun xctrace export` and parses its XML output. All version-specific logic lives in **`src/engine/versionRules.ts`** — a single file of data tables with no executable logic beyond lookups. The parser and engine (`src/engine/xctrace.ts`, `src/engine/session.ts`) are intentionally version-unaware; they read whatever XML the fixtures contain. Tests live in `tests/` and are driven by snapshot fixtures in `tests/fixtures/xcode-<version>/`. The open_trace tool returns a `versionWarning` block when it encounters an Xcode version not listed in VERSION_BASE — that warning is the user's signal to run this workflow.

---

## Files you will touch

| File | What it contains |
|------|-----------------|
| `src/engine/versionRules.ts` | `VERSION_BASE`, `VERSION_SCHEMA_OVERRIDES`, `VERIFIED_PAIRS` — the only place version gates live |
| `tests/fixtures/xcode-<version>/schema-table/*.xml` | One XML file per schema, exported from a real .trace |
| `tests/fixtures/xcode-<version>/track-detail/*.xml` | Track-detail fixtures (Allocations, Leaks) |
| `tests/__snapshots__/*.snap` | Auto-generated by `npm test -- -u`; commit them |
| `scripts/generate-fixtures.sh` | Script that runs the xcrun export commands |
| `src/lenses/` | Lens code — only touched in PR 2 of Scenario C |

**Do not touch** `src/engine/xctrace.ts`, `src/engine/session.ts`, `src/core/`, or `src/index.ts` unless the project owner has asked you to.

---

## Why .trace files are not in the repo

A recorded `.trace` bundle is 100–600 MB of binary data. Committing one would make the repo unusable. Instead, fixtures are the XML output of `xcrun xctrace export`, which is typically 2–50 KB per schema. These small XML files are what the tests load.

## Generating fixtures from a local .trace file

The script at `scripts/generate-fixtures.sh` runs all the necessary `xcrun xctrace export` calls for you:

```bash
./scripts/generate-fixtures.sh <xcode-version> <directory-containing-your-traces>
```

Example — Xcode 28.0, traces stored in `~/Documents/traces/`:

```bash
./scripts/generate-fixtures.sh 28.0 ~/Documents/traces
```

The script expects these trace files in the directory (create only what you need for your schemas):

| File | Schemas inside |
|------|---------------|
| `model.trace` | ModelInferenceTable, ModelLoadingTable, SessionTable, ToolTable, FMEventTable, RequestTable, InstructionsTable |
| `HangsAndHitches.trace` | hitches, potential-hangs, hang-risks |
| `network.trace` | network-connection-detected, NetworkConnectionStats, network-connection-update |
| `SwiftData.trace` | core-data-save, core-data-fetch, core-data-fault, core-data-relationship-fault |
| `swift.trace` | SwiftTaskLifetime, SwiftTaskStateTable, SwiftTasksInfoTable |
| `swiftUI.trace` | swiftui-updates, swiftui-causes, swiftui-changes, swiftui-layout-updates |
| `AllocAndLeaksWithBacktraces.trace` | Leaks (track-detail) |

Output lands in `tests/fixtures/xcode-<version>/schema-table/` and `tests/fixtures/xcode-<version>/track-detail/`.

### Check sizes before exporting

`.trace` bundles store data in a compact binary format. `xcrun xctrace export` decompresses and converts to verbose XML, which can be **10–100× the binary size** — a 300 MB trace file can produce gigabytes of XML for per-event schemas. Before exporting any schema:

1. **Inspect the TOC first** to see what schemas exist:
   ```bash
   xcrun xctrace export --input my.trace --toc
   ```

2. **Export to a temp file and check the size** before committing:
   ```bash
   xcrun xctrace export \
     --input my.trace \
     --xpath '/trace-toc/run[@number="1"]/data/table[@schema="swiftui-updates"]' \
     > /tmp/check.xml
   wc -c /tmp/check.xml   # anything over ~500KB needs to become a synthetic fixture
   ```

3. **Schemas that are almost always too large** for a real fixture:
   - `swiftui-updates`, `swiftui-causes`, `swiftui-changes` — record every attribute graph event; easily 4–365 MB
   - `time-sample` — one row per CPU sample at ~1 kHz; easily 7 MB for a 30-second recording
   - `Allocations List` — one row per live allocation; easily 50–80 MB

   For these, export the XML once to see the column schema, then build a **synthetic fixture** with 3–4 hand-written rows that cover the column types. See the xcode-27.0 fixtures for examples.

If you need to export a single schema manually (e.g. to iterate on one schema without re-running everything):

```bash
# schema-table format
xcrun xctrace export \
  --input ~/Documents/traces/HangsAndHitches.trace \
  --xpath '/trace-toc/run[@number="1"]/data/table[@schema="hitches"]' \
  > tests/fixtures/xcode-28.0/schema-table/hitches.xml

# track-detail format (Allocations, Leaks)
xcrun xctrace export \
  --input ~/Documents/traces/AllocAndLeaksWithBacktraces.trace \
  --xpath '/trace-toc/run[@number="1"]/tracks/track[@name="Leaks"]/details/detail[@name="Leaks"]' \
  > tests/fixtures/xcode-28.0/track-detail/Leaks__Leaks.xml
```

> **Filename convention for track-detail:** `{TrackName}__{DetailName}.xml` — two underscores encode the `/`; spaces in the detail name become `-`.

### Review before committing

Some schemas contain sensitive content. Before committing any fixture:

- **Foundation Models tables** (ModelInferenceTable, InstructionsTable, FMEventTable, SessionTable, RequestTable): may contain real user prompts and AI responses. Replace real rows with synthetic equivalents that preserve the column structure.
- **NetworkConnectionStats**: may contain real process names and IP addresses. Replace with neutral values.
- **SwiftUI tables** (swiftui-updates, swiftui-causes, swiftui-changes): may contain app-specific view type names, environment object names, and private service class names. Replace with generic SwiftUI framework type names. These fixtures are also too large to store — always create synthetic replacements.
- **time-sample / Allocations List**: can be 7–80 MB. Use a very short recording or create a synthetic fixture with a few representative rows.

## Scenario A — New Xcode version, existing schemas unchanged

**One PR:** `Adding compatibility for {xcodeVersion} {InstrumentName}`

The schemas and column mnemonics haven't changed, but the Xcode version number is new and the server is emitting a `versionWarning`. This is purely a data update.

### Steps

**1. Create a branch**

```bash
git checkout -b adding-compatibility-28.0-AllSchemas
```

(Use one branch per instrument if schemas differ; use one branch for all if nothing changed.)

**2. Add the version to VERSION_BASE**

In `src/engine/versionRules.ts`, add an entry to `VERSION_BASE`:

```typescript
export const VERSION_BASE: Record<string, string> = {
  "27.0": "27.0",
  "28.0": "27.0",  // inherits 27.0 rules — no format changes detected
};
```

If the Xcode version is truly format-identical to an existing one, point it at the existing rules version. If you found any column changes, use `VERSION_SCHEMA_OVERRIDES` for the affected schemas (see Scenario A with changes below).

**3. Generate fixtures**

```bash
./scripts/generate-fixtures.sh 28.0 ~/Documents/traces
```

Review all output for sensitive content (see above).

**4. Add the new (rulesVersion, schema) pairs to VERIFIED_PAIRS**

In `src/engine/versionRules.ts`, add one entry per fixture you exported:

```typescript
const VERIFIED_PAIRS = new Set<string>([
  // existing entries …
  "28.0:hitches",
  "28.0:potential-hangs",
  // etc. — one entry per XML file you committed
]);
```

**5. Examine every schema you exported**

Open each XML fixture and read the `<schema>` element. For every `<col>`, note:
- Its `engineering-type`
- Whether it carries signal the current column-role set doesn't capture
- Whether any `engineering-type` is new compared to the 27.0 fixture

Record these observations in your PR description. You are not required to act on them in this PR, but the observations inform future lens work.

**6. Run tests and baseline**

```bash
npm test          # should pass; new fixtures auto-load
npm test -- -u    # regenerate snapshots for new fixtures
npm test          # verify green
```

**7. Commit and push**

```bash
# One commit per instrument — never bundle multiple instruments in one commit
git add tests/fixtures/xcode-28.0/schema-table/hitches.xml \
        tests/fixtures/xcode-28.0/schema-table/potential-hangs.xml \
        tests/fixtures/xcode-28.0/schema-table/hang-risks.xml \
        src/engine/versionRules.ts \
        tests/__snapshots__/
git commit -m "Adding compatibility for 28.0 HangsAndHitches"

git push -u origin adding-compatibility-28.0-AllSchemas
gh pr create \
  --title "Adding compatibility for 28.0 HangsAndHitches" \
  --body "..."
```

**If schemas changed between versions**, add the affected schemas to `VERSION_SCHEMA_OVERRIDES` rather than creating a new base version:

```typescript
export const VERSION_SCHEMA_OVERRIDES: Record<string, Record<string, string>> = {
  "28.0": {
    "hitches": "28.0",  // column format changed; all other schemas still use 27.0 rules
  },
};
```

Then add a `28.0:hitches` entry to `VERIFIED_PAIRS`. **Stop here** — do not touch the parser. If the parser needs to change to handle the new format, raise it with the project owner before proceeding.

## Scenario B — New schema on an already-supported Xcode version

**One PR:** `Adding compatibility for {xcodeVersion} {InstrumentName}`

A schema exists in a trace but has no fixture yet. The universal verbs (`query`, `aggregate`, `find`) work on it immediately once a fixture and snapshot are in place.

### Steps

**1. Create a branch**

```bash
git checkout -b adding-compatibility-27.0-CoreData
```

**2. Export the fixture — check size first**

Export to a temp file and check the size before writing to the fixtures directory. Schemas that capture per-event data (SwiftUI updates, cause graphs, time samples, network connections) can produce XML that is 10–100× the binary trace size:

```bash
xcrun xctrace export \
  --input ~/Documents/traces/SwiftData.trace \
  --xpath '/trace-toc/run[@number="1"]/data/table[@schema="core-data-save"]' \
  > /tmp/check.xml
wc -c /tmp/check.xml
```

- **Under ~500 KB:** copy to `tests/fixtures/xcode-27.0/schema-table/core-data-save.xml` and proceed.
- **Over ~500 KB:** build a synthetic fixture instead. Copy the `<schema>` element from the XML (column definitions only), add 3–4 hand-written rows that cover the column types, and use `"MyApp"` with a fake PID for any process references. See `tests/fixtures/xcode-27.0/schema-table/swiftui-updates.xml` or `time-sample.xml` for examples.

Review for sensitive content (process names, IP addresses, user prompts, private class/type names from the app).

**3. Add to VERIFIED_PAIRS**

```typescript
"27.0:core-data-save",
```

**4. Examine the schema**

Read every `<col>` and `engineering-type`. Note anything interesting in the PR description.

**5. Run tests and baseline**

```bash
npm test -- -u
npm test
```

**6. Commit — one instrument per commit**

```bash
git add tests/fixtures/xcode-27.0/schema-table/core-data-save.xml \
        src/engine/versionRules.ts \
        tests/__snapshots__/
git commit -m "Adding compatibility for 27.0 CoreData"
```

## Scenario C — New instrument with a curated lens

**Two separate PRs in order.**

### PR 1 — Compatibility (mechanical)

`Adding compatibility for {xcodeVersion} {InstrumentName}`

Follow Scenario B above exactly. No lens code. The deliverable is: fixture XML, VERIFIED_PAIRS entry, passing snapshot tests.

The schema review is especially important here. Read every `<col>` and `engineering-type` in the exported XML, and write a section in the PR description titled "Schema observations" that notes:

- Which columns carry the primary signal (timing, count, weight)
- Any `engineering-type` values that are new or unusual
- Columns that look useful but don't map to an existing role
- Your hypothesis about what "suspicious" output looks like for this instrument

This section is the brief for PR 2. A PR 2 without a PR 1 schema analysis will be asked to go back and write one.

### PR 2 — Diagnostics / curated lens (requires domain knowledge)

`Improve diagnostics for {xcodeVersion} {InstrumentName}`

**Before writing any code, verify you can answer all of these:**

1. What does this instrument measure, in one sentence?
2. Which column(s) carry the primary signal that indicates a problem?
3. What does "suspicious" output look like — high values, long durations, specific states?
4. What is the analysis heuristic? ("Flag any row where X > threshold Y" is sufficient; vague heuristics produce wrong lenses.)

**If you can't answer these, stop and ask the user.** The user recorded the trace because they understand the domain. A lens that confidently surfaces the wrong signal is worse than no lens at all.

**Verify column claims against a real recording, not `--show-recording-options` or the schema name.** That command only lists configurable option keys, not the actual column/schema shape — record a real trace and read the TOC/export before asserting what a schema does or doesn't carry. Guessing from a name or a template's options JSON has produced real, shipped-then-caught mistakes in this project (an instrument assumed to have no diagnostic value on its own, based on the pattern of similar-looking instruments, turned out to carry its own resolved backtrace once actually recorded).

Once you can answer all four, add the lens in `src/lenses/`. Read `aidocs/howHintsWork.md` first — it covers the established design patterns for `nextActions`/`quickStart` (auto-derived vs. curated hints, table-wide vs. per-row, proactive `RECORDING_INTENTS` notes vs. reactive lens hints, when a hint belongs in a core verb instead of a lens) so a new lens follows the same shape as the existing ones rather than reinventing it. The PR description must include a "Heuristic" section that justifies why the lens surfaces the right signal.

## PR checklist (all scenarios)

Before opening a PR, verify:

- [ ] One instrument per commit — no bundled fixtures across multiple instruments
- [ ] Commit message follows convention: `Adding compatibility for {xcodeVersion} {InstrumentName}` or `Improve diagnostics for {xcodeVersion} {InstrumentName}`
- [ ] Every schema you claim to support has a fixture XML in `tests/fixtures/xcode-<version>/`
- [ ] Every new fixture has a corresponding entry in `VERIFIED_PAIRS`
- [ ] `npm test` passes with no failures
- [ ] Snapshots were regenerated (`npm test -- -u`) and the updated `.snap` file is committed
- [ ] Fixture files were reviewed for sensitive content (prompts, real process names, IP addresses, real PIDs)
- [ ] PR description includes "Schema observations" for every new or changed schema
- [ ] No changes to `src/engine/xctrace.ts`, `src/engine/session.ts`, `src/core/`, or `src/index.ts` (unless pre-approved)
- [ ] For lens PRs (PR 2): PR description includes a "Heuristic" section explaining what the lens surfaces and why

A PR claiming support for Xcode X with only partial schema coverage will be asked to explain or fill the gaps before merge.

## After the PR is merged — running your fork as your local MCP server

While your PR is under review (or if you want to use the updated support immediately), you can point your AI client at your local fork instead of the published package.

**Build your fork:**

```bash
npm install
npm run build
```

**Wire it into Claude Code** (replaces the published server for this project):

```bash
claude mcp add instruments-mcp-server -- node /absolute/path/to/your/fork/dist/index.js
```

**Or in Claude Desktop** (`~/Library/Application Support/Claude/claude_desktop_config.json`):

```json
{
  "mcpServers": {
    "instruments-mcp-server": {
      "command": "node",
      "args": ["/absolute/path/to/your/fork/dist/index.js"]
    }
  }
}
```

Restart the client after updating the config. You can now open your trace — `open_trace` will no longer emit a `versionWarning` for your Xcode version, and all the schemas you added fixtures for will resolve with `confidence: "verified"`.

To enable session call logging (useful for debugging which tools the agent is selecting):

```bash
# Claude Code — add --log to the args
claude mcp add instruments-mcp-server -- node /path/to/fork/dist/index.js --log
# Logs appear in ~/Library/Logs/instruments-mcp-server/session-<timestamp>.jsonl
```
